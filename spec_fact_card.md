# Clarify Board — Specification Draft v0.1

## Overview

A digital implementation of Kurpatov's methodology. The system allows users to externalize their thinking onto a single-screen canvas, where AI assists in structuring, connecting, and revealing blind spots.

**Core principle:** One screen = one problem. No scrolling. Visual density reflects mental complexity.

---

## 1. Card

A card is the atomic unit of the system — a single fact, pain, resource, or hypothesis.

### 1.1 Required Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `id` | string | Unique identifier (uuid) |
| `text` | string | The fact itself. Short, concrete. Max ~100 chars |
| `type` | enum | Category: `question`, `fact`, `pain`, `resource`, `hypothesis` |
| `embedding` | float[N] | Semantic vector (generated by AI from text) |
| `created_at` | timestamp | When the card was created |
| `is_root` | bool | If true, this is the central question. Always at center (0.5, 0.5). Only one per canvas |

### 1.2 AI-Generated Parameters

These are set by AI when card is created, but can be adjusted:

| Parameter | Type | Description |
|-----------|------|-------------|
| `emoji` | string | 1-2 emoji that capture the essence. AI chooses |
| `color` | string | Hex color. Derived from type, but AI can adjust shade/intensity |
| `importance` | float 0-1 | How central this fact is. Affects size |
| `confidence` | float 0-1 | How certain is this a fact vs speculation |

### 1.3 Layout Parameters (computed by engine)

| Parameter | Type | Description |
|-----------|------|-------------|
| `x`, `y` | float | Current position on canvas |
| `target_x`, `target_y` | float | Position where embedding "wants" to be |
| `size` | float | Visual size (derived from importance + connections count) |
| `z_index` | int | Layer for overlapping cards |

### 1.4 User-Adjustable

| Parameter | Type | Description |
|-----------|------|-------------|
| `pinned` | bool | If true, card stays where user placed it |
| `collapsed` | bool | Part of a stack, only header visible |
| `is_new` | bool | Temporary flag for newly added card (for highlight animation) |

### 1.5 Position Persistence

**Important UX principle:** When user drags a card, it becomes `pinned=true` and its position is saved. This ensures:
- Adding new cards does NOT shift existing cards
- User's manual arrangement is preserved
- Only unpinned cards respond to layout engine forces

## 2. Connection

A connection represents a relationship between two cards.

### 2.1 Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `id` | string | Unique identifier |
| `from_id` | string | Source card id |
| `to_id` | string | Target card id |
| `type` | enum | `causes`, `relates`, `contradicts`, `blocks` |
| `strength` | float 0-1 | How strong the connection. Affects spring force |
| `label` | string? | Optional short label on the line |
| `created_by` | enum | `ai`, `user` |

### 2.2 Connection Types

| Type | Visual | Meaning |
|------|--------|---------|
| `causes` | solid arrow → | A leads to B |
| `relates` | dashed line --- | A and B are connected somehow |
| `contradicts` | red line ✕ | A and B are in tension |
| `blocks` | thick line ▌ | A prevents B |

## 3. Canvas

The canvas is the single-screen workspace.

### 3.1 Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `question` | string | The central question/problem |
| `cards` | Card[] | All cards on canvas |
| `connections` | Connection[] | All connections |
| `bounds` | {w, h} | Fixed to viewport size |

### 3.2 Semantic Zones (optional, AI-generated)

AI can define "zones" based on embedding clusters:

```
zones: [
  { id: "finances", center: [0.2, 0.3], radius: 0.15, label: "Деньги" },
  { id: "relationships", center: [0.7, 0.4], radius: 0.2, label: "Отношения" },
]
```

These are soft — cards gravitate toward them but aren't locked.

## 4. Layout Engine

The engine computes card positions based on multiple forces.

### 4.1 Forces (in priority order)

1. **Pinning force** (strongest)
   - If `pinned=true`, card stays at user-set position

2. **Connection spring force** (strong)
   - Connected cards attract each other
   - Force proportional to `connection.strength`

3. **Embedding gravity** (medium)
   - Card drifts toward `target_x, target_y` (from embedding projection)

4. **Collision repulsion** (close range)
   - Cards push away when overlapping
   - Allows partial overlap but prevents full stacking

5. **Boundary containment** (hard)
   - Cards cannot leave screen bounds

### 4.2 Position Update

```
new_position = current_position + Σ(forces) × damping
```

Animation should be smooth — cards "float" to their positions over 300-500ms.

### 4.3 Embedding → Position Mapping

1. Collect all card embeddings
2. Apply UMAP/t-SNE to project to 2D
3. Normalize to canvas bounds [0,1] × [0,1]
4. This becomes `target_x, target_y`

Re-computed when new cards added. Existing cards smoothly transition to new targets.

## 5. AI Role

AI participates in the session as an active assistant.

### 5.1 Card Creation

When user inputs text:
1. AI validates: is this a fact or abstraction?
2. If abstraction → AI asks clarifying question
3. If fact → AI generates:
   - `embedding`
   - `type` (suggests, user can override)
   - `emoji`
   - `color` shade
   - `importance` estimate

### 5.2 Connection Discovery

AI continuously analyzes cards and:
- Proposes connections (user can accept/reject)
- Explains reasoning: "Кредит может вызывать бессонницу — связать?"

### 5.3 Gap Detection

AI monitors for:
- Empty zones (areas where embedding space suggests content but no cards exist)
- Missing links (cards that "should" be connected based on semantics)
- Isolated cards (no connections — is this intentional?)

AI asks probing questions:
> "Вижу много фактов про работу, но ничего про здоровье. Это осознанный выбор?"

### 5.4 Insight Surfacing

When patterns emerge, AI highlights:
- Центры тяжести (cards with many connections)
- Chains (A → B → C → D)
- Contradictions (cards that conflict)

## 6. Visual Style

### 6.1 Card Appearance

- Rounded rectangle with slight shadow
- Slight random rotation (±3°) for "handmade" feel
- Color: solid background from type, with AI-adjusted intensity
- Emoji in top-left corner
- Text centered
- Size varies with importance (1x to 1.5x base size)

### 6.2 Color Palette (defaults)

| Type | Base Color | Meaning |
|------|------------|---------|
| `fact` | #3B82F6 (blue) | Verified, concrete |
| `pain` | #EF4444 (red) | Problem, tension, fear |
| `resource` | #22C55E (green) | Asset, helper, opportunity |
| `hypothesis` | #EAB308 (yellow) | Unverified, assumption |

AI can adjust saturation/brightness based on `confidence`:
- High confidence → vivid
- Low confidence → muted/pastel

### 6.3 New Card Highlight

When a card is just added (`is_new=true`):
- **Pulse animation**: card gently pulses 2-3 times to draw attention
- **Glow effect**: soft colored glow around the card (matching its type color)
- **Duration**: highlight lasts ~3 seconds, then fades to normal state
- **Purpose**: helps user see where the new card appeared on the canvas

### 6.4 Connection Appearance

- Curved bezier lines (not straight)
- Thickness from `strength`
- Arrows for directional connections
- Subtle animation (slight pulse) for newly created

### 6.4 Density Behavior

As canvas fills:
- Cards slightly shrink to fit
- Overlapping allowed (z-index matters)
- Stacks form automatically when many cards in same area

---

## 7. Session Flow

1. **Start**: Empty canvas, question in center
2. **Dump**: User speaks/types facts, AI creates cards
3. **Organize**: Cards float to positions, user can adjust
4. **Connect**: AI proposes connections, user confirms/rejects
5. **Probe**: AI asks about gaps and blind spots
6. **Insight**: User sees patterns, has "aha" moment
7. **Action**: User identifies concrete next steps
